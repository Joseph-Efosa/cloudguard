# CloudGuard

**Automated GDPR Compliance Monitoring for Hybrid Multi-Cloud Environments (AWS + GCP)**

CloudGuard is an open-source Python tool that continuously evaluates your AWS and Google Cloud Platform configurations against 50 GDPR-derived technical controls drawn from Articles 5, 25, 30, and 32. It produces structured JSON and styled HTML compliance reports, and optionally enriches them with plain-language explanations and remediation guidance generated by a locally-hosted LLM (Llama 3 via Ollama).

> Developed as an MSc Cybersecurity research project at Technological University Dublin.

---

## Features

- **50 GDPR controls** mapped to AWS (C-01–C-25) and GCP (C-26–C-50)
- **Deterministic pass/fail evaluation** — no probabilistic guessing; every decision is traced to a specific API attribute
- **Dual-CSP unified scan** — assess AWS and GCP in one command
- **Ground truth evaluation** — compute Precision, Recall, F1-score, and False Positive Rate against a known-good JSON file
- **LLM remediation guidance** — locally-hosted Llama 3 (via Ollama) generates plain-English remediation steps per failed control
- **Terraform IaC** — reproducible compliant and deliberately-misconfigured test environments for both CSPs
- **CI/CD ready** — `--fail-on-violations` exits with code 1 if any FAIL is detected

## Installation

```bash
# 1. Clone the repository
git clone https://github.com/Joseph-Efosa/cloudguard.git
cd cloudguard

# 2. Create and activate a Python virtual environment
python3.11 -m venv .venv
```

## Credentials Setup

CloudGuard reads credentials from the standard locations for each cloud provider.
**Never hard-code credentials in code or commit them to version control.**

### AWS

CloudGuard uses `boto3`, which reads credentials from `~/.aws/credentials` or environment variables.

**Option A — AWS CLI (recommended):**
```bash
aws configure
# Enter your AWS Access Key ID, Secret Access Key, and default region (e.g. eu-west-1)
```

**Option B — Environment variables:**
```bash
export AWS_ACCESS_KEY_ID="your-access-key-id"
export AWS_SECRET_ACCESS_KEY="your-secret-access-key"
export AWS_DEFAULT_REGION="eu-west-1"
```

**Required IAM permissions:**

The AWS identity must have read-only access to the following services:
`s3`, `cloudtrail`, `rds`, `kms`, `iam`, `ec2`, `config`

A minimal policy is:
```json
{
  "Effect": "Allow",
  "Action": [
    "s3:GetBucketEncryption", "s3:GetBucketPublicAccessBlock", "s3:GetBucketVersioning", "s3:ListAllMyBuckets",
    "cloudtrail:DescribeTrails", "cloudtrail:GetTrailStatus",
    "rds:DescribeDBInstances",
    "kms:ListKeys", "kms:GetKeyRotationStatus", "kms:DescribeKey",
    "iam:GenerateCredentialReport", "iam:GetCredentialReport",
    "ec2:GetEbsEncryptionByDefault",
    "config:DescribeConfigurationRecorderStatus"
  ],
  "Resource": "*"
}
```

### GCP

CloudGuard uses Application Default Credentials (ADC).

**Option A — User credentials (development):**
```bash
gcloud auth application-default login
```

**Option B — Service account (production/CI):**
```bash
export GOOGLE_APPLICATION_CREDENTIALS="/path/to/service-account.json"
```

**Required IAM roles** (on the target project):

| Role | Purpose |
|---|---|
| `roles/storage.admin` (or `roles/storage.objectViewer`) | Read bucket metadata |
| `roles/logging.viewer` | Read audit log configuration |
| `roles/cloudsql.viewer` | Read Cloud SQL instance settings |
| `roles/iam.securityReviewer` | Read IAM policy bindings |
| `roles/compute.viewer` | Read disk encryption configuration |
| `roles/secretmanager.viewer` | Read secret metadata |

A single custom role combining `*.get` and `*.list` permissions on all services is sufficient.

## Ollama / LLM Setup (Optional)

CloudGuard's HTML report optionally includes per-control explanations and remediation guidance generated by Llama 3 running locally via Ollama. This keeps all configuration data on your machine.

```bash
# 1. Install Ollama from https://ollama.com

# 2. Pull the Llama 3 8B model (approx. 4.7 GB download)
ollama pull llama3

# 3. Verify Ollama is running
curl http://localhost:11434/api/tags
```

The scanner works fully without Ollama — reports are still generated using the deterministic evaluation output.

## Usage

### Full dual-CSP scan

Scans all 20 controls across both AWS and GCP simultaneously.

```bash
python main.py --gcp-project YOUR_GCP_PROJECT_ID
```

*Replace `YOUR_GCP_PROJECT_ID` with your GCP project ID (e.g. `my-gdpr-project-2026`).*

The AWS region defaults to `eu-west-1`. To change it:

```bash
python main.py --gcp-project YOUR_GCP_PROJECT_ID --aws-region eu-central-1
```

---

### AWS only

Scans only the 25 AWS controls (C-01 through C-25). No GCP project required.

```bash
python main.py --csp aws

# With a non-default region
python main.py --csp aws --aws-region us-east-1
```

---

### GCP only

Scans only the 25 GCP controls (C-26 through C-50). No AWS credentials required.

```bash
python main.py --csp gcp --gcp-project YOUR_GCP_PROJECT_ID
```

---

### Run specific controls

Pass one or more control IDs to scan only those checks.

```bash
# Scan a single control
python main.py --controls C-01

# Scan multiple controls (any combination of AWS and GCP)
python main.py --controls C-01 C-02 C-03 \
    --gcp-project YOUR_GCP_PROJECT_ID

# Scan GCP encryption controls
python main.py --controls C-26 C-34 C-36 \
    --csp gcp --gcp-project YOUR_GCP_PROJECT_ID

# Scan all audit-logging-related controls
python main.py --controls C-03 C-04 C-10 C-28 C-29 C-30 C-42 \
    --gcp-project YOUR_GCP_PROJECT_ID

# Scan all Article 32 controls on AWS
python main.py --controls C-01 C-05 C-06 C-07 C-09 C-15 C-18 C-23 \
    --csp aws
```

---

### With ground truth evaluation

Provide a JSON file mapping each control ID to its expected state (`true` = compliant, `false` = misconfigured). CloudGuard will compute Precision, Recall, F1-score, and False Positive Rate.

```bash
# Evaluate against a fully-compliant environment
python main.py \
    --gcp-project YOUR_GCP_PROJECT_ID \
    --ground-truth ground_truth_compliant.json

# Evaluate against a fully-misconfigured environment
python main.py \
    --gcp-project YOUR_GCP_PROJECT_ID \
    --ground-truth ground_truth_misconfigured.json
```

**Ground truth file format (`ground_truth_compliant.json`):**
```json
{
  "C-01": true,
  "C-02": true,
  "C-03": true,
  "C-26": true,
  "C-27": true
}
```

`true` means the control is expected to PASS (environment is compliant).
`false` means the control is expected to FAIL (environment is misconfigured).

---

### LLM-enriched HTML report

Run the LLM reporter directly on an existing results JSON to add Ollama-generated explanations and remediation guidance.

```bash
# Generate the all-fail LLM report (synthetic — no live cloud scan required)
python generate_misconfigured_report.py
# Output: cloudguard/reports/report_misconfigured_all_fail.html

# Serve the HTML report locally to view in a browser
python -m http.server 8765 --directory cloudguard/reports
# Then open: http://localhost:8765/report_misconfigured_all_fail.html
```

---

### CI/CD pipeline gate

Exit with code 1 if any FAIL result is detected. Use this to fail a pipeline build when GDPR violations are introduced.

```bash
python main.py \
    --csp aws \
    --ground-truth ground_truth_compliant.json \
    --fail-on-violations

echo "Exit code: $?"   # 0 = all pass, 1 = violations found
```

**GitHub Actions example:**
```yaml
- name: GDPR compliance check
  run: |
    source .venv/bin/activate
    python main.py --csp aws --fail-on-violations
  env:
    AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
    AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
    AWS_DEFAULT_REGION: eu-west-1
```

---

### Suppress verbose output

```bash
python main.py --csp aws --quiet
```

---

### Custom controls file

Point to your own YAML control catalogue instead of the default one:

```bash
python main.py --controls-file /path/to/my_controls.yaml --csp aws
```

---

### Custom output directory

```bash
python main.py --csp aws --output-dir /tmp/cloudguard-results
```

### Synthetic all-fail report

Generate an HTML report with Ollama remediation text for all 20 controls in FAIL state — no live AWS or GCP scan required. Useful for reviewing what the LLM guidance looks like before running against real infrastructure.

```bash
python generate_misconfigured_report.py
```

Output is written to `cloudguard/reports/report_misconfigured_all_fail.html`.

## CLI Reference

```
usage: main.py [-h] [--gcp-project GCP_PROJECT] [--aws-region AWS_REGION]
               [--csp {aws,gcp}] [--controls ID [ID ...]]
               [--controls-file PATH] [--output-dir PATH]
               [--ground-truth PATH] [--fail-on-violations] [--quiet]
```

| Argument | Default | Description |
|---|---|---|
| `--gcp-project` | `$CLOUDGUARD_GCP_PROJECT` | GCP project ID to scan |
| `--aws-region` | `eu-west-1` | AWS region to scan |
| `--csp` | *(both)* | Limit scan to `aws` or `gcp` |
| `--controls` | *(all applicable)* | Space-separated list of control IDs to run (e.g. `C-01 C-05`) |
| `--controls-file` | `controls/gdpr_controls.yaml` | Path to custom YAML control catalogue |
| `--output-dir` | `reports/` | Directory where JSON and HTML reports are written |
| `--ground-truth` | *(none)* | JSON file for evaluation metrics (P/R/F1/FPR) |
| `--fail-on-violations` | *(off)* | Exit with code 1 if any FAIL result found (CI/CD use) |
| `--quiet` | *(off)* | Suppress per-control INFO log messages |

**Environment variables** (alternative to flags):

| Variable | Equivalent flag |
|---|---|
| `CLOUDGUARD_GCP_PROJECT` | `--gcp-project` |
| `CLOUDGUARD_AWS_REGION` | `--aws-region` |

## Terraform Test Infrastructure

The `terraform/` directory contains reproducible IaC definitions for both AWS and GCP environments used in the research evaluation. Each configuration accepts a boolean variable `var.misconfigured`:

- `false` (default) — provisions all resources in a **GDPR-compliant** state
- `true` — provisions resources with **deliberate misconfigurations** matching the controls under test

### Provision a compliant AWS environment

```bash
cd terraform/aws

# Initialise Terraform providers
terraform init

# Preview what will be created
terraform plan

# Apply (creates compliant resources)
terraform apply

# When finished — destroy all resources to avoid charges
terraform destroy
```

### Provision a misconfigured AWS environment

```bash
cd terraform/aws
terraform init
terraform apply -var="misconfigured=true"
# Run CloudGuard scan, then clean up:
terraform destroy -var="misconfigured=true"
```

### Provision a compliant GCP environment

```bash
cd terraform/gcp
terraform init
terraform apply -var="project_id=YOUR_GCP_PROJECT_ID"
```

### Provision a misconfigured GCP environment

```bash
cd terraform/gcp
terraform init
terraform apply \
    -var="project_id=YOUR_GCP_PROJECT_ID" \
    -var="misconfigured=true"

# Clean up
terraform destroy \
    -var="project_id=YOUR_GCP_PROJECT_ID" \
    -var="misconfigured=true"
```

> **Cost warning:** The Terraform configurations provision real cloud resources that may incur charges.
> Always run `terraform destroy` when you have finished testing.

## Evaluation Methodology

CloudGuard was evaluated against 7 scenarios across both CSPs:

| Scenario | Description | Controls | Expected Violations |
|---|---|---|---|
| S1 | AWS Compliant | C-01–C-25 | 0 of 25 |
| S2 | AWS Misconfigured | C-01–C-25 | 25 of 25 |
| S3 | GCP Compliant | C-26–C-50 | 0 of 25 |
| S4 | GCP Misconfigured | C-26–C-50 | 25 of 25 |
| S5 | Dual-CSP Compliant | C-01–C-50 | 0 of 50 |
| S6 | Dual-CSP Misconfigured | C-01–C-50 | 50 of 50 |
| S7 | Mixed (AWS✓, GCP✗) | C-01–C-50 | 25 of 50 |

**Results across all 350 control evaluations (7 × 50):**

| Metric | Score |
|---|---|
| Precision | 1.00 |
| Recall | 1.00 |
| F1-Score | 1.00 |
| False Positive Rate | 0.00 |
